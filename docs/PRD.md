# Fitness AI Assistant · 产品需求文档 (PRD)

> 文档版本：v0.1（架构规划阶段）
> 最后更新：2026-05-18
> 项目定位：**个人练手 demo**，目标是把一个"AI 驱动的健身助手"以生产级工程化标准跑起来。**不上架商店、不做商业合规**，但代码与架构按真实项目要求写。

---

## 1. 项目愿景

一个面向**力量训练 + 体重/体型管理**人群的移动端助手，把"训练记录—计划—饮食—反馈"这个闭环用 AI 串起来：

- 用户随手拍饭菜 → AI 给出热量与宏量营养（蛋白/碳水/脂肪）
- 用户告诉 AI 当前的力量水平、身体状况、目标 → AI 生成可执行的 4 周训练 + 饮食方案
- 用户每天打卡 → 计划周期结束后 AI 复盘并调整
- （Phase 2）用户上传体检报告 → AI 提取关键指标并修正计划
- （Phase 2）社区分享打卡 / 食谱 / 训练照片，找训练伙伴

## 4. 核心用户旅程

### 4.1 新用户首次进入

```
启动 → 手机号+密码注册 → 完善档案（性别/年龄/身高/体重/训练年限）→
力量水平登记（卧推/深蹲/硬拉 1RM，可跳过）→ 选择目标（增肌/减脂/维持）→
AI 生成首份 4 周训练计划 + 饮食计划 → 进入主页
```

### 4.2 老用户日常使用

```
打开 app → 主页看今日训练 → 完成训练 → 逐组打卡（含组间计时）→
午餐拍照识别 → 记入饮食日志 → 晚上看仪表盘（训练完成度 / 今日热量 / 本周趋势）
```

### 4.3 周期复盘

```
4 周 mesocycle 结束 → 后台 worker 触发复盘任务 →
AI 分析打卡完成率 / 重量进步 / 体重变化 →
推送站内通知 → 用户进入复盘页 → 选择"按建议调整"或"自定义" →
生成下一个 4 周计划
```

## 5. 关键业务规则

### 5.1 训练相关

- **力量水平**通过 1RM 表征。允许用户跳过，AI 用经验公式给保守初值（基于性别/体重/训练年限）
- **训练计划**最小单位是"训练日"，一个 mesocycle = 4 周 = 4 个微周期，每个微周期 3–6 个训练日
- **打卡可偏离计划**：实际做的组数/重量与计划允许不一致，记 `plannedSet` 与 `actualSet` 双轨数据
- **离线打卡**：草稿存在 RN 端 MMKV，恢复网络时由 TanStack Query 的 mutation 队列上传

### 5.2 饮食相关

- 一切重量、热量、营养素**仅用公制**（g / kcal / ml）
- 食物识别命中本地库时使用库内数据，未命中走 VLM 估算并标记 `source: AI_ESTIMATE`
- 每日热量目标基于 TDEE × 目标系数（增肌 +10%、减脂 −20%、维持 ±0%）
- 宏量比例缺省：增肌 蛋白 30% / 碳水 45% / 脂肪 25%，减脂 蛋白 40% / 碳水 30% / 脂肪 30%

### 5.3 AI 调用

- 所有 AI 调用通过 BullMQ 异步任务执行，HTTP 接口立即返回 `taskId`
- 客户端通过 `GET /ai/tasks/:id` 轮询，或后续接 WebSocket 推送
- 所有 AI 调用产生一条 `ai_runs` 记录：`taskType / model / prompt / response / tokenIn / tokenOut / costCNY / status / errorMsg`
- 失败任务最多重试 3 次（指数退避），仍失败标 `FAILED` 并通知用户

### 5.4 数据建模通则

- 主键统一 `cuid2`
- 软删除字段 `deletedAt: DateTime?`
- 时间字段统一 UTC 存储，前端按设备时区展示
- 所有用户产生的内容关联 `userId` 并加索引

## 6. 非功能性需求

| 维度     | 目标                                                                |
| -------- | ------------------------------------------------------------------- |
| 启动时间 | RN 应用冷启动 < 3s（中端 Android）                                  |
| API 响应 | 普通查询 P95 < 300ms（本地开发），AI 任务异步无 SLA                 |
| 离线     | 训练打卡 100% 离线可用，恢复网络自动同步                            |
| 错误监控 | 客户端 Sentry，后端 pino → 本地文件                                 |
| 数据安全 | 密码 argon2 哈希；refresh token 存 Keychain；TLS（本地开发可 http） |
| 可观测性 | AI 任务全量落 `ai_runs` 表，含成本统计                              |
| 可移植性 | StorageProvider / LLMProvider / OAuthProvider 三个接口可换实现      |

## 7. 风险与边界

| 风险                   | 处理                                                                       |
| ---------------------- | -------------------------------------------------------------------------- |
| AI 输出不结构化 / 幻觉 | 全量 Zod schema 校验 + 失败重试 + 提供"手动编辑"兜底                       |
| VLM 识别准确率低       | 标记 `AI_ESTIMATE` + 用户可一键修正 + 修正样本沉淀（Phase 2 训练自有模型） |
| 训练动作存在版权       | 仅记录文本，不内置视频/图，预留 `mediaUrl` 字段后期手动补                  |
| 数据库 schema 变化频繁 | 严格走 Prisma migration；MVP 阶段允许 `prisma migrate reset`               |
| 计划生成成本飙升       | 单用户每日 AI 任务限额 5 次（DB 校验），超限直接拒绝                       |
| 健康数据隐私           | Demo 不做应用层加密；架构上预留列级加密 hook（`@Encrypted` 装饰器）        |

## 8. 验收标准（MVP）

一个新用户能在 5 分钟内完成：

1. 注册并完成档案
2. 看到 AI 为自己生成的 4 周训练计划与今日饮食计划
3. 拍一张饭菜照片，得到热量估算并自动入账
4. 完成一次训练打卡，仪表盘正确反映完成度

所有 P0 功能在 Android 8.0+ 真机上完整跑通。
